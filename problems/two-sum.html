Two Sum

LeetCode Arrays & Hashing All Problems
Problem

Given an array of integers nums and an integer target, return indices of the two numbers such that they add up to target.

You may assume that each input would have exactly one solution, and you may not use the same element twice. You can return the answer in any order.

Constraints
  • 2 <= nums.length <= 104
  • -109 <= nums[i] <= 109
  • -109 <= target <= 109
  • Only one valid answer exists.
Examples
Example 1
Input: nums = [2,7,11,15], target = 9 Output: [0, 1]
nums[0] + nums[1] = 2 + 7 = 9, so return [0, 1].
Example 2
Input: nums = [3,2,4], target = 6 Output: [1, 2]
Example 3
Input: nums = [3,3], target = 6 Output: [0, 1]
Interview Tips
▶ How to ace this in an interview Expand
🗣 Talk through it

Read the problem aloud. Restate it in your own words. Confirm your understanding with the interviewer before writing any code.

❓ Clarifying questions (and why we ask each)

What's the size of the array?
Why: Determines acceptable complexity. n ≤ 10⁴ → O(n²) brute force is OK. n ≤ 10⁶ → need O(n) or O(n log n). The constraint dictates the algorithm.
Are negative numbers, zeros, or duplicates allowed?
Why: Negatives are fine for hash-map (any int can be a key) but break some range-based tricks. Zeros mean a value can be its own complement (target=0, value=0). Duplicates mean the same value can appear at different indices — the answer might pair two equal numbers (e.g., [3,3] target 6).
Is exactly one valid answer guaranteed, or could there be zero / many?
Why: If guaranteed unique, you can return on the first match. If multiple are valid, the question changes ("any one" vs "all pairs"). If none guaranteed, you need a sentinel return value — usually an empty list or [-1, -1].
Can I modify the input array?
Why: Some algorithms (sort-then-two-pointer) mutate. Interviewers may expect immutability for code-cleanliness reasons. Always ask — costs nothing, prevents wasted work if they say "no".
What should I return — indices, values, or a boolean?
Why: Two Sum returns indices, but the variants don't. "Two Sum II" returns 1-indexed positions. "Two Sum IV — Input is a BST" returns a boolean. Misreading the return spec is a classic interview face-plant.
🧠 Think out loud

Start with brute force — nested loops, O(n²) time. Estimate its complexity and say it. Then explain how to do better: "I can trade space for time using a hash map." State the tradeoff explicitly. Implement the simpler-to-explain approach first, not the cleverest. Test on the given example before touching edge cases.

⚠ Common mistakes
  • Using the same element twice (e.g. [3,3], target=6 is fine, but don't reuse a single element).
  • Off-by-one when returning indices — double-check on Example 2.
  • Hashing the values array instead of tracking indices — you need to return positions, not values.

🎯 How to think about edge cases

Before you say "I'm done," walk through these categories. For each, ask: does my code handle this gracefully?

Minimum-size input
Constraint says n ≥ 2. Test [1,2] with target 3 — does it return [0,1]?
Duplicates (same value, different indices)
Test [3,3] with target 6 — your hash map must NOT block the second 3 from finding the first. Most off-by-one bugs hide here.
Negative numbers and zero
Test [-1, 1, 0] with target 0. Does complement = target - x work when target is 0 or negative? It should — but verify.
Solution at the boundaries
What if the answer is the first two elements (e.g., [1,2,...] target 3) or the last two ([...,9,10] target 19)? Walk through both mentally.
Single-element edge: same element twice
Constraint says "may not use the same element twice." A hash map naturally enforces this if you check the map BEFORE inserting the current value. Confirm your ordering.
What's the answer if no pair exists?
Problem says exactly one solution is guaranteed, so this shouldn't happen. But if I asked you to extend the problem to handle "no answer", what would you return? Knowing the answer signals seniority.

General framework: empty / minimum / maximum input, extreme values (min/max int, negative, zero), duplicates, solution-at-boundary, no-solution, multiple-solutions, ordering matters?, mutability allowed?

🏢 How FAANG interviews this expand for grading rubric + how to ace it

📋 How they introduce it

"Here's a warm-up. Given an array of integers and a target, return the indices of the two numbers that add up to target. Each input has exactly one solution, and you can't use the same element twice."

Often the first question in a 45-min loop, or a warm-up before a harder follow-up (3Sum, 4Sum, Two Sum II).

🎯 What they're really testing

  • Can you turn a vague prompt into a clear contract by asking the right clarifying questions?
  • Do you instinctively think in complexity tradeoffs (brute force → optimization)?
  • Can you write clean code under mild pressure (naming, structure, no off-by-ones)?
  • Do you test your own code on the given example without being asked?
  • Can you narrate your thought process — "think out loud"?

📊 The unofficial grading rubric

Strong Hire
Asked 3+ clarifying Qs without prompting. Stated brute force first, immediately proposed hash-map optimization. Walked through the example before AND after coding. Clean code with semantic names. Discussed edge cases (duplicates, negatives) unprompted. Total time: ~10 min.
Hire
Reached the optimal O(n) solution. May have skipped 1 clarifying question or 1 edge case. Code is correct but might have a minor stylistic issue. Total time: 15-20 min.
Lean No-Hire
Jumped to coding without asking questions. Confused indices vs values mid-solution. Needed a nudge to think about the hash map. Tested only after being asked.
No-Hire
Couldn't reach O(n) even with hints. Couldn't trace through an example. Wrote buggy code and didn't recognize the bugs when reading their own code. Stayed silent for long stretches.

🎓 How to ace it (the 8-step playbook)

  1. Restate the problem in your own words. ("I need indices i, j with nums[i] + nums[j] = target, can't reuse the same index. Right?")
  2. Ask 2-3 clarifying questions from the list in Interview Tips above. Don't ask 10 — interviewers find that excessive.
  3. Propose brute force out loud. "Simplest: check every pair. O(n²) time, O(1) space." Don't write it — just state it.
  4. State the optimization. "I think we can do O(n) with a hash map. Tradeoff: O(n) extra space."
  5. Walk through ONE example before coding. Show how the hash map fills up and detects the pair. This proves to the interviewer you actually understand it.
  6. Code cleanly. Variable names: seen, complement. Not d, x, tmp.
  7. Test on the given example AFTER coding. Walk through it. Confirm output matches expected.
  8. Mention edge cases you'd handle if pushed: duplicates (e.g., [3,3]), no-answer (return what?), large n. Interviewer may probe one.

Total time: 10-15 min for a strong run. Finish faster and they'll ask a follow-up (3Sum, sorted array variant, return all pairs, etc.) — be ready.

🚫 What NOT to do

  • Don't code immediately. A pause to think is mature, not weak.
  • Don't overoptimize prematurely. Don't try sorting + binary search — go straight to hash map.
  • Don't return values instead of indices. Read the spec twice.
  • Don't forget the "same element" rule. nums[i] + nums[i] is NOT valid.
  • Don't go silent when stuck. Narrate your thinking — interviewers can't help you if they don't know where you are.
  • Don't argue with the interviewer. If they say "what if there are duplicates?", explore it. They're checking how you handle pushback.

Solution Walkthroughs
Brute Force Time: O(n²) Space: O(1) 
def two_sum(nums, target):
    for i in range(len(nums)):
        for j in range(i + 1, len(nums)):
            if nums[i] + nums[j] == target:
                return [i, j]

Check every possible pair. For each element at index i, scan all elements after it to find a complement. This is the first thing you'd say in an interview — it shows you understand the problem before jumping to the optimal approach. Slow, but provably correct. O(n²) time because of the nested loop; O(1) space because you only store loop indices.

Hash Map (Optimal) Time: O(n) Space: O(n) 
def two_sum(nums, target):
    seen = {}
    for i, x in enumerate(nums):
        complement = target - x
        if complement in seen:
            return [seen[complement], i]
        seen[x] = i

For each number x, ask: "have I already seen target - x?" If yes, we have our pair — return its stored index and the current index. If no, store x with its index for future lookups. A single pass through the array, O(n) time. We trade O(n) space (the hash map) for the speed gain. The key insight: instead of looking forward for a pair, we look backward at what we've already seen.
